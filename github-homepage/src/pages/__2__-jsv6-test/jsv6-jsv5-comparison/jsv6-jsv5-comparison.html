4. 认证与安全模型

维度 v5（GMS-exhibition） v6（jsv6-test）
前端暴露的凭证 client-id 写在 script src URL 里，任何人打开 DevTools 即可看到 仅暴露短效 clientToken（服务端生成，有效期短，泄露风险低）
client-id 位置 前端 URL 参数（公开） 仅存在于后端环境变量，前端完全不感知
clientToken 获取 无此概念 页面加载时调用后端接口 /api/paypal/client-token 获取，再传入 SDK
Partner Attribution（BN Code） 通过 Header PayPal-Partner-Attribution-Id 传递，支持 partner 归因统计 测试项目中未配置，实际集成时应通过 SDK 初始化选项传入
后端鉴权方式 Bearer Token（OAuth2 client_credentials） Basic Auth（base64(clientId:secret)），适合服务端直接调用

5. 支付方式的声明与加载方式

支付方式 v5（GMS-exhibition） v6（jsv6-test）
PayPal 标准钱包 paypal.Buttons() 默认渲染，无需额外声明 实例化时声明 components: ["paypal-payments"]
BCDC（基础卡直接输入） fundingSource: paypal.FUNDING.CARD + expandCardForm: true，内联展开卡表单 独立实例：components: ["paypal-guest-payments"]，创建 createPayPalGuestOneTimePaymentSession
ACDC（高级卡 / 3DS） 同一 SDK 实例内，通过 fundingSource: FUNDING.CARD 并在 order 中传 payment_source.card 触发 独立路由 + 独立 create-order 端点（create-order-ACDC）
BNPL / Pay Later URL 参数 enable-funding=paylater 开启，fundingSource: FUNDING.PAYLATER 渲染按钮 独立测试页面；SDK 层面通过 components 声明控制
国家 / 币种变更 需销毁并重新加载整个 SDK 脚本（URL 参数 buyer-country 与 currency 绑定在加载时） SDK 本身无感知，仅需切换后端 create-order 端点（如 /create-order-EUR）
是否支持 isEligible 检查 有 button.isEligible()，不满足条件时可跳过渲染 SDK 内部处理，开发者无需手动检查 eligibility

6. 核心设计哲学总结

SDK v6 的设计方向

  • Run-time 配置，不依赖 URL 参数
  • client-id 完全隐藏在后端，安全性更高
  • 开发者持有 PaymentSession 引用，完全控制触发时机
  • presentationMode 显式控制弹窗 / 重定向行为
  • 多币种 / 多国家切换无需重载 SDK
  • 与 Stripe、Apple Pay、Google Pay 等保持相近的 API 规范

SDK v5 的设计方向

  • Load-time 配置，所有参数在 script URL 中声明
  • client-id 前端可见，接入简单但安全级别较低
  • SDK 自动接管完整 UI 流程，开发者干预空间有限
  • isEligible() 机制让开发者可根据环境条件控制渲染
  • 支持 partner 归因（BN Code / Auth Assertion）
  • 快速集成首选，适合对安全要求不高的内部测试环境
